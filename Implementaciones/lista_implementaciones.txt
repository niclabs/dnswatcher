1. Disponibilidad por versión de IP (IPv4/IPv6)
Estado: Implementado
Referencia RSSAC047: 5.1 - RSI Availability
Descripción: Evalúa si los servidores autoritativos de cada dominio responden correctamente bajo ambas versiones de protocolo IP.
¿Cómo?: SE realizan consultas tipo A (IPv4) y AAAA (IPv6) sobre los NS de cada dominio. Se cuenta cuántos responden por cada protocolo.


2. Disponibilidad por tipo de transporte (UDP/TCP)
Estado: En revisión
Referencia RSSAC047:  5.1 - RSI Availability
Descripción: Determina si los servidores soportan consultas DNS a través de UDP y TCP, fundamentales para garantizar compatibilidad y tolerancia a fallos.
¿Cómo?: Se consulta a cada IP asociada a NS mediante tipo A usando ambos transportes. Se cuenta cuántos responden exitosamente.


3. Latencia de respuesta y *latencia de publicación
Estado: Latencia de respuesta: Implementada.
        Latencia de publicación: Pendiente, se implementará usando la infraestructura oficial del Observatorio con claves geoid.
Referencias RSSAC047: 5.2 - RSI Response Latency
                      5.4 - RSI Publication Latency
Descripción: Latencia de respuesta: mide el tiempo que demora en recibirse una respuesta a una consulta SOA, evaluando si cumple con los umbrales recomendados (≤250 ms UDP, ≤500 ms TCP).
             Latencia de publicación: evaluará cuánto tiempo tarda un vantage point en observar un nuevo serial SOA propagado en toda la red. Este monitoreo se realizará con vantage points identificados mediante claves geoid, integrados con la implementación oficial del Observatorio.
¿Cómo?: Latencia de respuesta: Se realizan consultas tipo SOA al dominio raíz (“.”), registrando los tiempos de respuesta por transporte. Se calculan medianas y se comparan con los umbrales del paper.
        Latencia de publicación: (Pendiente) Se desarrollará en coordinación con la infraestructura del Observatorio, aprovechando la identificación geográfica de vantage points mediante claves geoid para medir la propagación real de actualizaciones de seriales SOA.



4. Cumplimiento de estándares DNS y DNSSEC
Estado: Implementado -> al pasar a OBS hay que alinearlo con geolite
Referencia RSSAC047: 5.3 - RSI Correctness
Descripción: Evalúa si las respuestas de los servidores de nombres cumplen los estándares del root zone y están correctamente firmadas con DNSSEC.  
Se realizan consultas a registros clave (SOA, NS, DNSKEY) y consultas negativas (NXDOMAIN aleatorio). Para cada respuesta se revisa que incluya firmas RRSIG, registros NSEC/NSEC3 y que tenga las banderas correctas (AA).  
Actualmente se mide por IP y transporte (UDP/TCP) y se calcula el porcentaje de respuestas correctas.  
Esta parte se implementó como prueba para validar la lógica de verificación, base para la integración final en el Observatorio.


5. Tasa de éxito/fallo en DNSSEC
Estado: Implementado 
Referencia RSSAC047: 5.3 - RSI Correctness
Descripción: Cuantifica la proporción de respuestas DNSSEC correctamente validadas v/s fallidas.
Idea: - En las respuestas recibidas, revisaremos si están presentes los registros RRSIG, NSEC, NSEC3.
      - Con esto validamos la firma con dnskey obtenido de una consulta separada.
      - Cuentamos los casos con y sin validación exitosa -> calculamos porcentaje.


6. Validación de firma y registros DS
Estado: Implementado
Referencia RSSAC047: 5.3
